\## 1. DETECTION



\### 1.1 Structural Invariant Alerts

| Invariant | Alert Condition | Severity |

|-----------|-----------------|----------|

| #1 (Provider Registration) | Call chain mismatch | HIGH |

| #2 (Provider Table Integrity) | MOV/JMP trampoline detected | CRITICAL |

| #3 (Event Write Call Site) | Patched call site | HIGH |

| #6 (Kernel Callback) | Unhooking attempt | CRITICAL |



\### 1.2 Behavioral Analytics (Charlotte AI)

\- Sudden drop in ETW events from a provider (>50%)

\- Unusual process creating ETW sessions (e.g., not svchost.exe)

\- High-frequency provider registration/unregistration



\### 1.3 EDR Telemetry (Falcon / MDE)

\- Alert: "ETW provider tampering detected"

\- Alert: "Kernel callback modification"

\- Event ID 4688 with suspicious command line (e.g., `powershell -Version 2`)



\## 2. CONTAINMENT



\### 2.1 Immediate Actions

1\. \*\*Isolate affected host\*\* from network (air-gap if possible).

2\. \*\*Capture memory dump\*\* for forensic analysis.

3\. \*\*Preserve ETW logs\*\* (event tracing sessions).



\### 2.2 Short-Term Mitigation

\- If tampering is ongoing, \*\*terminate the offending process\*\*.

\- \*\*Block outbound DNS\*\* to known malicious domains (see threat intel report).

\- \*\*Disable PowerShell v2.0\*\* if downgrade attack is suspected.



\## 3. ERADICATION



\### 3.1 Root Cause Analysis

\- Identify how the attacker gained initial access (phishing? vulnerability?).

\- Check for persistence mechanisms (scheduled tasks, registry run keys).

\- Review EDR logs for lateral movement.



\### 3.2 Remediation

\- \*\*Patch\*\* any exploited vulnerabilities.

\- \*\*Reset credentials\*\* for affected accounts.

\- \*\*Remove\*\* any backdoors or implants.



\## 4. RECOVERY



\### 4.1 Restoration

\- Reimage host from known-good backup (air-gapped source).

\- Re-enable network connectivity only after verification.

\- Monitor for recurrence (continuous monitoring framework).



\### 4.2 Validation

\- Run full structural invariant test suite.

\- Verify detection accuracy against a simulated attack.

\- Update threat model if new variant discovered.



\## 5. POST-INCIDENT



\### 5.1 Reporting

\- File incident report using template (`docs/templates/incident-report-template.md`).

\- Share intelligence with threat intel team for APT FENRIR tracking.



\### 5.2 Lessons Learned

\- Update detection logic if gaps identified.

\- Adjust training materials (see `docs/training/intro-to-invariants.md`).

\- Recalibrate thresholds if drift detected (Step 5-6).
